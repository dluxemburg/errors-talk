- Errors get confusing because JavaScript is _pretty_ asynchronous - (or at least more confusing that it might be in other language)
Callbacks Promises Events Streams
Good Errors? ❌
Loud Failures? ❌
- We've got callbacks as a way to handle errors that come out of asynchronous behavior - ...but they've got problems with _seemingly_ inaccurate messages - ...and silent failures - ...they're also not composable, resulting in redundant handling
Callbacks Promises Events Streams
Good Errors? ❌ ✅
Loud Failures? ❌ ❌
- As compared with callbacks... - ...promises give us better errors because they drop us into a "`catch`" block immediately instead of letting us wander off to try to do something else in an unexpected state - ...but they've still got issues with failing silently unless we're proactive about handling - (it's a different type of silent failure, true, but still bad!) - (the global handler isn't really a solution because if your code is being used by someone else they might have their own ideas about what to do at the top level, better to keep things local) - ...but they're nicely composable!
Callbacks Promises Events Streams
Good Errors? ❌ ✅ ✅
Loud Failures? ❌ ❌ ✅
- So events... - ...also give us better errors because, like promises, we're immediately bounced into targeted error handling code - ...but unlike promises, if that handlier isn't there the error automatically jumps up to the global context so it can't be ignored - ...but they're not composable in the way promises are - we don't get to collect errors from different chuncks of code (event if we can do things in parallel)

So...

  • Callbacks: Most adaptable—good when handling depends on context

  • Promises: Most composable—good when errors from steps can be aggregated

And then but so...

  • Events: Most loud—good when things should blow up if an error isn't handled

  • Streams: Most niche—like events but with an interface for combining
